# Stakeholder Register

Living document — one row per stakeholder, added as they are identified (not only after they've been interviewed). `STK-###` IDs are permanent and never reused, like all IDs in this repo.

| ID | Name | Role / department | Interest | Influence | Preferred channel | Sessions (TRN) |
|---|---|---|---|---|---|---|
| STK-001 *(example — delete)* | Maria Chen | Customer Support Team Lead | Owns reset-ticket queue; wants agent time back | high | Email, weekly 1:1 | TRN-001 |
|  |  |  |  |  |  |  |

**Interest** — what they want from (or fear about) the project, in their words where possible.
**Influence** — `high | medium | low`: their power to change scope, budget, or adoption.

## Coverage Check

Before requirements sign-off, ask: is every affected department represented by at least one row with a session logged? A register full of `high` influence rows and empty `Sessions` cells is an elicitation plan, not evidence.
